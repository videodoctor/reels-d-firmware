**Date range:** January 18–22, 2026

## Summary

Over ~4 sessions, successfully ported 0dan0's Type C firmware modifications through Phase 8 to the Type D hardware. This is the first working D-variant firmware modification in the community.

**Stable result:** 1600×1200 resolution at 18fps on Type D hardware (vs. stock 1728×1296 at 20fps)

---

## Key Discoveries

### The +0x1164 Offset
Discovered a consistent offset of **+0x1164** between most Type C and Type D function addresses. This was found by comparing Ghidra-exported symbol tables using Python binary diffing scripts. Not universal — exceptions exist in the new sensor pipeline code.

### Function Entry Hook Safety Rule
**Phase 8 initially hooked at `0x2bfeb8` (a function entry point) — this caused the motor to continuously advance.**

Root cause: On Type D, hooking at a function entry point interferes with the capture control flow in a way that doesn't happen on Type C. The fix was to hook inside the function instead. This is a D-specific behavior and a critical rule for all future hook work.

### Ghidra Setup
- Used Ghidra headless initially but it hung on analysis errors (jumps/branches outside loaded range)
- Switched to Ghidra GUI via X11 forwarding (MobaXterm → AWS Ubuntu instance)
- Both firmwares loaded as Raw Binary, MIPS LE 32-bit, base address `0x80000000`
- This setup works well and should be the standard approach

## Phase 9 Attempt — FPS Control (FAILED)

**Goal:** Change default frame rate from 20fps to 18fps on Type D.

**Approach:** In Type C, this is a simple patch at `0x1ef984` — change value to `0xF0D2` for 18fps. Applied offset to get D candidate address and patched.

**Result:** Capture loop freeze. Unit stops responding during capture. Requires power cycle.

**Conclusion:** The FPS/timing control is one of the exceptions to the +0x1164 offset rule. The new D sensor likely has a different clock/timing register structure. **Do not re-attempt without new Ghidra analysis.** See `sensor-notes.md` for the investigation approach.
